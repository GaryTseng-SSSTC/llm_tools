正式環境

作業系統 Ubuntu 22.04.3 LTS

測試環境建置目標

  • 完整載入Llama 3.3 70B-AWQ模型（約35GB）
  • 提供completion API給其他系統使用
  • 不考慮TTFS等效能指標
  • 測試環境與正式環境一致性最大化

軟硬體方案比較

四種主要方案的詳細分析與比較

A

方案A：單卡NVIDIA 48GB（推薦）

最貼近正式環境

硬體配置

  • GPU: NVIDIA RTX 6000 Ada 48GB 或 A6000 48GB
  • CPU: 16實體核心級（Intel Xeon/W 或 AMD EPYC）
  • RAM: 64GB（建議）
  • 儲存: NVMe SSD 1TB
  • 電源: ≥1000W，白金等級

優勢與風險

優點：
  • • 與正式環境GPU廠牌一致
  • • 單卡顯存餘裕>8GB
  • • 移轉風險最低
風險：
  • • 需注意KV cache顯存限制
  • • 成本相對較高
B

方案B：雙卡NVIDIA 24GB×2

張量平行，成本敏感

硬體配置

  • GPU: 2×RTX 4090（或3090/4080 Super）
  • 主機板: 需2組x16或x16+x8 PCIe
  • RAM: 64GB（建議）
  • 電源: ≥1600W（雙4090供電）
  • 啟動: --tensor-parallel-size 2

優勢與風險

優點：
  • • 總顯存48GB，透過分片載入
  • • 成本通常低於單張專業卡
風險：
  • • 4090無NVLink，跨卡通訊靠PCIe
  • • 驅動/NCCL設置複雜
C

方案C：AMD GPU（ROCm）

不建議，除非策略要求

硬體配置

  • GPU: AMD Instinct MI210 64GB、MI300X 192GB
  • 軟體: Ubuntu 22.04.3 + ROCm 6.x
  • 框架: PyTorch(ROCm) + vLLM(ROCm build)
  • 驗證: AWQ量化路徑與HIP/ROCm kernel

風險評估

主要風險：
  • • 與正式環境CUDA棧差異大
  • • ROCm框架成熟度相對分散
  • • AWQ kernel最佳化程度未知
  • • 功能回歸測試工作量大
D

方案D：NVIDIA NIM微服務

策略備選

特色說明

  • 核心: 預先建置、最佳化的推論微服務
  • 內含: CUDA、Triton、TensorRT-LLM
  • 部署: 快速上線，維運簡化
  • 基礎架構: 仍需NVIDIA GPU（48GB建議）

一致性影響

差異風險：
  • • 與現行vLLM推論引擎不同
  • • API/行為細節可能有差異
  • • 資源管理方式不同
  • • 需評估未來正式環境遷移計畫

軟體配置

作業系統 Ubuntu 22.04.3 LTS
容器 Docker 28.1.1
推論框架 vllm-openai:v0.9.1
模型 llama-3.3-70b-instruct-awq

部署與驗證指令

單卡啟動（方案A）：

docker run --gpus all --rm -p 8000:8000 --shm-size 16g \
  -e HF_TOKEN=${HF_TOKEN} \
  vllm/vllm-openai:v0.9.1 \
  --model casperhansen/llama-3.3-70b-instruct-awq \
  --quantization awq \
  --dtype bfloat16 \
  --gpu-memory-utilization 0.90 \
  --max-num-batched-tokens 8192 \
  --max-num-seqs 32

雙卡分片（方案B）加入：

--tensor-parallel-size 2

API服務說明：

vLLM提供OpenAI風格的Completions/Chat API伺服器（/v1/completions, /v1/chat/completions）， 可直接供既有系統串接。使用--quantization awq可啟用4-bit量化以降低顯存需求； 上述配置在實務上可將70B級模型顯存壓到~40GB內。

AMD GPU與NVIDIA NIM說明

主管關注的兩個特殊方案詳細分析

AMD GPU方案

可行性評估

  • • 單卡建議至少48GB顯存（MI210 64GB、MI300X 192GB）
  • • 需ROCm 6.x、PyTorch(ROCm)與vLLM的ROCm版本
  • • 重新驗證AWQ在HIP/ROCm的相容性與穩定性

一致性風險

  • • 與正式環境CUDA棧差異大
  • • 框架/驅動/Kernel最佳化差異
  • • 測試結果難以無縫推廣至正式環境

建議方向

若組織有「轉向AMD」之中長期策略，可建立「平行測試線」做可用性驗證； 短期若目標是與正式一致，仍建議選NVIDIA。

NVIDIA NIM方案

核心特色

  • • 預先建置、最佳化的推論微服務容器
  • • 內含CUDA、Triton Inference Server與TensorRT‑LLM
  • • 快速部署於各式NVIDIA加速基礎架構

一致性影響

  • • 與現行vLLM推論引擎不同
  • • API/行為細節可能有差異
  • • 資源管理與最佳化策略不同

使用時機建議

若組織有計畫於未來正式環境導入NIM，則可建立「NIM測試線」並行評估； 若近期不更動正式環境，建議測試仍以vLLM為主。

風險與緩解措施

潛在風險點與對應的解決方案

顯存不足/OOM

載入模型時可能出現記憶體不足錯誤

緩解措施：

  • • 限制--max-num-seqs參數
  • • 調整--max-num-batched-tokens
  • • 預留顯存餘裕（單卡48GB建議）

模型授權合規

Llama 3.3使用需遵循授權條款

緩解措施：

  • • 採用社群商用授權
  • • 依授權條款內部測試使用
  • • 確保合規使用範圍

軟體版本漂移

測試與正式環境版本不一致風險

緩解措施：

  • • 固定Ubuntu 22.04.3版本
  • • 鎖定Docker 28.1.1
  • • 使用vllm-openai:v0.9.1

多卡通訊（方案B）

雙卡方案的驅動與通訊穩定性

緩解措施：

  • • 確認驅動/NCCL穩定性
  • • 驗證PCIe拓樸設定
  • • 接受PCIe帶寬限制
